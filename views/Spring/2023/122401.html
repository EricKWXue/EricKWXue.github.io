            开启多事务不当引发问题

            Eric Xue 2023-12-24 spring boot transaction

            # 1. 背景

            对外提供服务的Rest API上线以来，一直运行非常“稳定”。但从国庆假期前一个礼拜开始，接入该系统API的同仁偶有提单过来反馈说有数据丢失现象：前脚显示插入数据成功，后脚查询又显示“no data”。

            查询当天日志，找到调用的API没有任何相关的报错，暂无头绪。只能先手动补上丢失的数据，该问题留待观察。

            没想到的是！到了国庆放假期间，亚太/欧洲/美洲的同仁大量提单过来反馈该问题。心想完犊子了，好在能靠重启服务缓解（没错！每次重启都能缓解问题两天，就跟症状突然好了一样，但两天后又不行了，好玄学 =.=）

            熬过国庆后，仔仔细细排查了该问题，用时一个礼拜总算水落石出了。

            # 2. 现象

            • 主要现象一：插入数据的API返回成功，但是有时实际数据成功入库，有时数据却没入库。
            • 主要现象二：日志文件内出现频繁的报错信息
            Cause: java.sql.SQLException: connection disabled; uncategorized SQLException; SQL state [null]; error code [0]; connection disabled; nested exception is java.sql.SQLException: connection disabled
            1
            • 主要现象三：针对没有成功入库的API，查询skywalking调用链，发现最后的事务Tx没有调用commit和close方法。
            • 附加现象四：数据库表时不时出现长时间等待、或者直接锁表的情况。

            # 3. 排查过程

            1. 首先怀疑的是新提交的代码，找到最早发生该问题的那一周提交的代码，看到那周寥寥无几的几行代码，直接排除嫌疑。
            2. 接着又追踪了一遍API用到了声明式事务注解的地方，有人怀疑DAO层用了事务注解导致的问题（解释一下为什么DAO层会写事务注解，我们批量插入数据有时会有上万条，减小数据库缓存压力，在DAO层会切分成合理的条数多次执行插入，为了保证数据一致性所以需要事务注解），所以秉持怀疑一切心态先去掉了注解试试。事实证明，没卵用，问题依旧存在。
            3. 接着又怀疑druid连接池没有剔除失效的Connection，测试机模拟了一下失效的Connection（手动kill掉数据库session），事实再次证明druid没问题，又是一场误会。
            4. 实在没辙了，老板组织了一个小会召集相关同事一起看看目前情况。在汇报数据库锁表情况的时候，发现每次锁表锁住的第一张表永远是A表，很奇怪的是：这个A表跟发生数据丢失的API没有任何关系。我们找到涉及A表的API，排查了一遍可疑的代码，发现它用了一个自定义的多数据源事务注解，其他API没有使用过。所以再次秉持怀疑一切心态先去掉了试试，哦豁，问题消失了。

            多事务注解MultiTransactional，对应切面的源码

            @Aspect
public class TransactionAspect {

    @Around(value = "@annotation(multiTransactional)")
    public Object around(ProceedingJoinPoint point, MultiTransactional multiTransactional) throws Throwable {
        Stack<DataSourceTransactionManager> dataSourceTransactionManagerStack = new Stack<>();
        Stack<TransactionStatus> transactionStatuStack = new Stack<>();
        //开启事务
        if (!openTransaction(dataSourceTransactionManagerStack, transactionStatuStack, multiTransactional)) {
            return null;
        }
        //执行业务方法
        Object ret = null;
        try {
            ret = point.proceed();
        } catch (Throwable e) {
            //回滚事务
            rollback(dataSourceTransactionManagerStack, transactionStatuStack);
            throw e;
        }
        //提交事务
        commit(dataSourceTransactionManagerStack, transactionStatuStack);
        return ret;
    }

    /**
     * 开启事务
     */
    private boolean openTransaction(Stack<DatasourceTransactionManager> datasourceTransactionManagerStack, Stack<TransactionStatus> transactionStatuStack, MultiTransactional multiTransactional) {
        //取得注解里多个数据源的名称
        String[] transactionMangerNames = multiTransactional.values();
        if (Arrayutil.isEmpty(multiTransactional.values())) {
            return false;
        }
        //依次开启事务，而后放入栈内
        for (String beanName : transactionMangerNames) {
            DatasourceTransactionManager dataSourceTransactionManager = (DatasourceTransactionManager) SpringUtil.getBean(beanName);
            TransactionStatus transactionStatus = dataSourceTransactionManager.getTransaction(new DefaultTransactionDefinition());
            transactionStatuStack.push(transactionStatus);
            datasourceTransactionManagerStack.push(dataSourceTransactionManager);
        }
        return true;
    }
}

            第9行调用开启事务方法，第38行在for循环内依次开启多个事务。多个事务放在同一个线程开启，如果不加异常处理，可能会有问题。

            此时假设A_DS、B_DS进来，需要先后开启事务。A_DS事务开启成功，它的ConnectionHolder就会绑定到ThreadLocal里（这个TheadLocal内部维护一个集合，可以存放多个数据源），紧接着B_DS事务开启失败，这时就会抛出运行时异常。

            因为没做异常处理，导致前面成功开启事务的A_DS连接一直没有发生rollback/commit（也就是未完成状态），也就没有从ThreadLocal解绑，最后该线程被回收到tomcat线程池里。

            当后面某个倒霉的用户请求（其他API），不小心被tomcat分配到了这个自带ConnectionHolder的线程，直接从ThreadLocal取到了之前的ConnectionHolder，因为最后插入数据时被判断出不是newTransaction，最终不会提交。所以就出现了数据丢失的问题，数据库执行语句也一直处于未提交状态。

            当一条sql插入语句长时间不提交，占着茅坑不拉屎，后面关于该表操作的sql可能就会发生等待。当越来越多的sql发生等待，直到DB锁住了，最后DBA不得已删除了所有session。此时该条DB的Connection会失效不可用，但是ConnectionHolder依旧绑在线程上（ConnectionHolder持有Connection对象），用到该线程的请求执行sql就会出现java.sql.SQLException: connection disabled

            至此，所有现象的原因都能解释清楚了。

            # 4.1 FQA

            • 为什么事务会开启失败？

            答：凌晨3点的数据库B业务比较繁忙，响应非常慢，开启事务时发生connection timeout了。

            • 为什么之前没有定位问题到这个API？

            答：该API早在一年前就上线了，但是用户迟迟没有接入，直到国庆前一个礼拜才偷偷摸摸接入，没有任何人告知我们这件事。

            # 4.2 补充Spring声明式事务逻辑

            主要逻辑都在TransactionAspectSupport类的invokeWithinTransaction方法里，大概逻辑如下：

            1. createTransactionIfNecessary()开启事务，保存事务信息到ThreadLocal。
            2. invocation.proceedWithInvocation()执行业务代码逻辑。
            3. 提交或者回滚事务：发生异常执行completeTransactionAfterThrowing(txInfo)，无异常执行commitTransactionAfterReturning(txInfo)。最后从ThreadLocal中剔除事务信息。

            # 6. 总结

            • 编程式事务（即手动操作事务）要格外小心！开启事务，执行方法，提交事务，回滚事务，最好都放在一个大的异常处理里，保险起见加个finally。
            @Around(value = "@annotation(multiTransactional)")
public Object around(ProceedingJoinPoint point, MultiTransactional multiTransactional) throws Throwable {
    Stack<DataSourceTransactionManager> dataSourceTransactionManagerStack = new Stack<>();
    Stack<TransactionStatus> transactionStatuStack = new Stack<>();
    try {
        //开启事务
        if (!openTransaction(dataSourceTransactionManagerStack, transactionStatuStack, multiTransactional)) {
            return null;
        }
        //执行业务方法
        Object ret = point.proceed();
        //提交事务
        commit(dataSourceTransactionManagerStack, transactionStatuStack);
        return ret;
    } catch (Throwable e) {
        //回滚事务
        rollback(dataSourceTransactionManagerStack, transactionStatuStack);
        throw e;
    }
}
